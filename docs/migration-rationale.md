# 変更理由（v1 → v2）

> v1 から v2 で技術選定を変えた理由をまとめたもの。各項目は「v1 の状況 → 課題 → v2 の選択と理由」で記述する。面接で「なぜ変えたのか」を聞かれた際の根拠として使える形にしている。
> ⚠️ 印の項目は実装方針が確定していない / 自分の実際の意図に合わせて要調整。

## 1. 通信方式：REST → GraphQL（graphql-ruby + Apollo Client）

- **v1 の状況**：Rails の REST API。`lib/*.ts` に fetch を手書きし、レスポンス型も `types/` に手書きしていた。画面ごとに必要なデータ形が違うため、`today_words` / `stats/total_words` / `public_wordbooks/:uuid/children` / `study_records/week` / `study_records/recent` のように**エンドポイントが増殖**していた。
- **課題**：
  - 画面が必要とする以上のデータを取る over-fetch、逆に複数回叩く under-fetch が起きやすい。
  - フロントとバックの型整合を**手動で**保つ必要があり、API 変更時に追従漏れが起きうる。
- **v2 の選択**：GraphQL に統一。クライアントが必要なフィールドだけを 1 リクエストで取得でき、エンドポイント増殖を抑えられる。`graphql-ruby` でスキーマを単一の真実とし、フロントの型は **GraphQL Codegen で自動生成**して乖離をビルド時に検出する。
- **トレードオフも理解**：GraphQL は N+1 を招きやすいので、サーバー側は **DataLoader / batch loading** で対策する。単純な CRUD だけなら REST で十分という前提も承知の上で、「画面ごとにデータ要件が異なる学習アプリ」という性質に GraphQL が合うと判断した。

## 2. データ取得・状態管理：TanStack React Query → Apollo Client

- **v1 の状況**：React Query を `queryKey` 設計・`enabled`・`invalidateQueries`・`setQueryData` まで活用できていた。一方で、fetch ロジック（`lib`）とキャッシュ設定（`hooks`）の**二層**を持ち、`queryKey` を手で管理していた。
- **課題**：GraphQL に移行すると、REST 前提の React Query では型生成やキャッシュ正規化の恩恵を十分に受けられない。
- **v2 の選択**：GraphQL とセットで Apollo Client に統一。**正規化キャッシュ**により ID 単位でのキャッシュ更新が自動化され、`queryKey` の手管理が不要になる。Codegen が型付きフックを生成するため、`lib` + `hooks` の二層を 1 つに集約できる。
- **補足**：React Query で身につけた「サーバーステートとクライアントステートの分離」「Mutation 後の再取得」という考え方はそのまま Apollo（`refetchQueries` / `cache.writeQuery`）に転用できる。

## 3. UI：自作コンポーネント + CSS Modules / Tailwind → MUI 7

- **v1 の状況**：Button / ButtonSecondary / FloatingInput / Card 系などを**自作**し、CSS Modules / Tailwind でスタイリングしていた。Composition や floating label を CSS だけで実装するなど、基礎は押さえていた。
- **課題**：デザインの一貫性・アクセシビリティ（focus / aria）・日付ピッカー等のリッチ部品を**すべて自前**で支える必要があり、画面が増えるほどコストが上がる。
- **v2 の選択**：MUI 7 を採用し、デザインシステム・アクセシビリティ・フォーム部品を標準化する。`x-date-pickers` など完成度の高い部品を使うことで実装速度を上げ、本質（学習ロジック）に集中できる。実務でも MUI 採用例が多い。
- **補足**：v1 で UI を自作した経験（責務分割・状態の hoisting・再利用設計）は無駄ではなく、MUI 上でも同じ設計指針を適用する。

## 4. リッチテキスト：プレーンテキスト → TipTap

- **v1 の状況**：単語帳の説明などはプレーンな文字列。
- **v2 の選択**：TipTap を導入し、説明やメモをリッチテキストで編集できるようにする。学習コンテンツの表現力を上げる。

## 5. 構成：個別構成 → モノレポ（packages/frontend + packages/backend）

- **v1 の状況**：frontend / backend がそれぞれ独立（別 README・別管理）。
- **課題**：GraphQL スキーマや Codegen 設定など、フロント・バックで共有したい資産が分散する。
- **v2 の選択**：Yarn workspaces のモノレポにまとめ、依存・CI・開発体験を統合する。スキーマ起点の型共有がやりやすくなる。

## 6. テスト：薄い構成 → RSpec（バックエンド）+ Playwright（E2E）

- **v1 の状況**：テストは最小限。
- **v2 の選択**：可読性の高い RSpec でモデル・ロジックを固め、Playwright で主要導線（ログイン → テスト → 記録）を E2E で守る。品質への意識を形にする。

## 7. ⚠️ 認証：JWT（ステートレス）→ DB セッション（activerecord-session_store）

> ※ tech-stack.md では DB セッション方式を採用。JWT を残す場合はこの項目を調整。

- **v1 の状況**：JWT によるステートレス認証（user_token / admin_token を Cookie 保存）。
- **JWT の課題（理由になりうる点）**：トークンが発行後に独立して有効なため、**即時失効（強制ログアウト・漏洩時の無効化）が難しい**。有効期限切れまで使えてしまう。
- **v2 の選択（採用する場合）**：DB セッションにすることで、サーバー側でセッションを**即時失効・一元管理**できる。学習アプリの規模ではステートレスの水平スケール優位性より、運用の制御性を優先できる。
- **トレードオフ**：DB セッションはサーバー状態を持つため、スケール時に共有ストアが要る。この点を理解したうえでの選択だと説明できると良い。

## 8. ⚠️ インフラ・DB：開発・本番ともに PostgreSQL に統一（デプロイは Render + Vercel を想定）

> ※ 当初は AWS + Kamal + MySQL 8 統一を想定していたが、現段階では **PostgreSQL 統一 + Render / Vercel** に方針変更。最終構成は調整中。

- **v1 の状況**：フロントは Vercel、バック・DB は Render の無料枠。開発は MySQL、本番は PostgreSQL と**環境で DB が分かれていた**。
- **課題**：
  - 開発と本番で DB が異なり、差異起因の不具合リスクがある。
  - Render 無料枠のスリープ等の制約。
- **v2 の選択**：開発・本番ともに **PostgreSQL 16 に統一**し環境差を解消。デプロイは当面 **Render（バックエンド）+ Vercel（フロント）** で軽量に立ち上げ、Render 純正のマネージド PostgreSQL を使う。Solid 系（Queue/Cache/Cable）も PostgreSQL に寄せ Redis 不使用を維持。将来的に Kamal + AWS への移行も選択肢として残す。
- **トレードオフ**：マネージド前提で自前運用の学びは減るが、初期は本番化までの速度とシンプルさを優先。フロント／バックが別ドメインになるため、認証（Cookie / トークン）と CORS の設計に注意が要る。

## 面接で伝えるときの指針

- 各変更を「**新しいから**」ではなく「**v1 で実際に困ったこと → それをどう解決するか**」で語る。上の「課題」欄がその核。
- v1 の良かった設計（権限分離・データ整合性の多層防御・ドメインロジックのモデル集約）は**捨てずに引き継ぐ**と明言する。技術を変えても設計思想は一貫している、という筋が一番強い。
- トレードオフ（GraphQL の N+1、DB セッションのスケール、自前インフラの運用負荷）に自分から触れると、「流行を追っただけではない」ことが伝わる。
