# 要件定義（MemoRise v2）

> v1 の実装から抽出した要件を、v2 の要件定義として再構成したもの。機能要件・ドメインモデルは v1 を踏襲し、技術スタックは [tech-stack.md](./tech-stack.md) の新構成（GraphQL / Apollo / MUI / Rails 8 Solid）を前提とする。

## 1. プロダクト概要

単語暗記学習の**継続**を支援する Web アプリ。単語帳管理・テスト・学習記録を通じて、「**学習の習慣化**」と「**進捗の可視化**」を実現する。公式単語帳によるすぐ始められる体験と、自作単語帳による自由な学習を両立させる。

### 解決する課題
- 単語帳をめくるだけでは達成感が得られず、学習が**習慣化しない**。
- 一度間違えた単語が積み上がらず、**苦手が放置される**。
- 自分の取り組み量が**可視化されず**、モチベーションが続かない。

### 目的（プロダクトの軸）
1. **間違えた単語を確実に復習につなげる** — テスト結果から誤答単語を自動で復習リストへ登録し、復習導線を整える。
2. **継続のモチベーションを生む** — 連続学習日数（streak）と学習カレンダーで「積み上がっている感覚」を提供する。
3. **すぐ始められる** — 公式単語帳をプリセットで用意し、自作の手間なく学習を開始できる。

## 2. ユーザーロールと権限

| ロール | 説明 | 主な権限 |
| --- | --- | --- |
| 未ログイン（public） | 認証前 | ログイン / 新規登録のみ |
| 準ログイン（semi-auth） | 一部機能を開放 | 公式単語帳の閲覧・テスト、今日の一問、マイページ閲覧 |
| 一般ユーザー（auth） | 認証必須 | 自作単語帳の作成・編集・削除、テスト、復習、学習記録の閲覧 |
| 管理者（admin） | 運営者 | 公式単語帳の作成・編集・削除・CSV 一括登録、ユーザー一覧閲覧、統計閲覧 |

- 一般ユーザーと管理者は**認証スコープを完全分離**する（別トークン・別エンドポイント・別画面空間）。一般ユーザーから管理者機能へはアクセスできない。
- 公式単語帳は一般ユーザーにとって**閲覧専用**。作成・編集・削除は管理者のみ。

## 3. 機能要件

### 3.1 トップ（ホーム）
- **今日の一問**：復習対象・公式単語からランダムに 1 問を表示し、単語と出会う回数を増やす。未取得時は内蔵のフォールバック単語を表示する。
- **公式単語集への導線**：自作不要ですぐ学習を始められる。
- **自作単語帳への導線**。
- **週ごとの streak 表示**：今週（月〜日）の学習状況を可視化してやる気を促す。
- **復習対象件数のバッジ表示**：「あと N 個復習」を常時表示し継続を促す。
- 未ログイン時は「ログイン / 新規登録」への導線（リッチな空状態）を表示する。

### 3.2 認証
- 新規登録（名前 / メール / パスワード）。
- ログイン / ログアウト。
- 一般ユーザーと管理者でログイン画面・トークンを分離。
- ユーザー作成時に「はじめての単語帳」を自動生成し、初回の空画面を防ぐ。

### 3.3 単語帳管理
- 公式単語帳の閲覧（親 = TOEIC / 英検 等、子 = 章 / Day の**親子階層**）。
- 自作単語帳の作成・編集・削除。
- 単語帳はレベル・ラベル（junior_high / eiken / toeic / official など）・パート（章）・並び順を持つ。
- 単語の追加・編集・削除。
- 削除は**論理削除**（学習履歴の参照切れを防ぐ）。

### 3.4 単語テスト
- 単語帳（子単語帳）ごとにテストを実行。出題順はシャッフル。
- 「答えを見る」→「正解 / 不正解」判定。答えを見るまで正誤判定は不可（UI 制約）。
- 正答率ゲージを表示。
- テスト終了時に**1 回だけ**学習記録を保存（履歴と進捗を同時更新、二重送信防止）。
- 誤答単語を自動で復習タグに登録する。

### 3.5 復習支援
- テストで間違えた単語を**復習タグ単語**として記録（単語帳横断で扱う独立リソース）。
- 復習専用テスト（`/wordbooks/review`）で誤答単語だけを再学習。

### 3.6 学習記録・可視化
- 日ごとの学習記録（解いた結果を保存。1 日 1 レコードに集約し、回数を累積）。
- カレンダー形式で学習状況を表示（月送り対応）。
- 週単位・直近の記録を別ビューで表示。

### 3.7 マイページ
- 連続学習日数（streak）の表示。
- 登録単語数の表示。
- プロフィール編集。

### 3.8 進捗（章の解放）
- 公式単語帳の章（パート）を**順番に解放**する。あるパートを完了すると次のパートが解放される。
- 解放状態は進捗レコードの存在で表現する。

### 3.9 管理者機能
- 公式単語帳の作成・編集・削除。
- 単語の**CSV 一括登録**（行番号付きエラーを返す）。
- ユーザー一覧の閲覧。
- 統計の閲覧。

## 4. 非機能要件

| 区分 | 要件 |
| --- | --- |
| セキュリティ | 認証スコープの分離。外部公開 ID は UUID（連番を露出せず推測アクセスを防止）。CORS は明示的ホワイトリスト。ログから機密パラメータ（パスワード・メール・トークン等）を除外。 |
| データ整合性 | DB・モデル・アプリ層の 3 段で整合性を担保（ユニーク制約・カスタムバリデーション・トランザクション）。 |
| パフォーマンス | 一覧表示の N+1 回避（counter_cache / eager loading）。集計系はキャッシュ活用。 |
| 国際化 / 表記 | 日本語ロケール対応。タイムゾーンは JST。週初めは月曜。 |
| 可用性 / 運用 | ヘルスチェックエンドポイント。Docker による開発環境統一。Kamal によるデプロイ。Lint / 脆弱性スキャンを CI に組み込む。 |
| SEO | メタデータ（title / description）設定、フォント最適化。 |

## 5. v1 からの主な変更方針（v2）

- **通信方式**：REST + 独自 fetch ラッパー → **GraphQL（Apollo Client / graphql-ruby）** に統一。型は GraphQL Codegen で自動生成。
- **状態管理**：TanStack React Query → **Apollo Client のキャッシュ**に集約。
- **UI**：自作コンポーネント + CSS Modules / Tailwind → **MUI 7** ベースのデザインシステム。
- **リッチ入力**：単語帳説明等に **TipTap** を導入可能に。
- **構成**：frontend / backend 個別構成 → **モノレポ（packages/frontend + packages/backend）**。
- ドメインモデル・学習フロー・権限設計といった**本質的な設計は v1 を踏襲**する（[backend.md](./backend.md) / [frontend.md](./frontend.md) 参照）。
